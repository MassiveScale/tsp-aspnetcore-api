# Output Cleaning

By default (`clean-output-dirs: true`) the emitter removes previously-generated
files from each output directory (models, interfaces, controllers, services,
helpers, and validators) **before** it writes new ones. This keeps the output in
sync with the spec: when you rename or delete a TypeSpec type, its old `.g.cs`
file is not left orphaned in the tree.

## Guardrails

Cleaning is intentionally conservative and never removes hand-written code. A
file is deleted only when **both** conditions hold:

1. Its name ends with the configured `file-extension` (default `.g.cs`), and
2. Its contents carry the `<auto-generated>` marker emitted by the default
   `file` template header.

Anything else is preserved:

- A file that matches the extension but lacks the marker is **left in place**
  and reported with an `output-file-left-behind` warning naming the file, so you
  can decide whether to remove it yourself. (When in doubt, the emitter keeps
  the file.)
- Files that do not match the generated extension (e.g. hand-written `.cs`) are
  ignored silently.

Directories are scanned recursively, so files placed in namespace-derived
sub-folders and per-version validator sub-directories are cleaned too. Empty
directories are left in place.

## Custom `file` templates

Cleaning relies on the `<auto-generated>` marker in the `file` template header.
If you override the `file` template (see [Custom Templates](custom-templates.md)),
keep that marker so generated files remain eligible for cleanup; otherwise set
`clean-output-dirs: false` and manage the directories yourself.

## Disabling

Set `clean-output-dirs: false` to disable cleanup entirely and only ever write
files.
